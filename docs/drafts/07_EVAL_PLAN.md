# Evaluation Plan

## 목적

Agent가 장기적으로 신뢰 가능한 개인 비서가 되려면 기능별 평가 기준이 필요하다.

MVP 완료/실패 기준은 [26_ACCEPTANCE_CRITERIA.md](26_ACCEPTANCE_CRITERIA.md)를 우선한다.

## 평가 영역

### 기억 품질

- 중요한 정보를 빠뜨리지 않는가.
- 불필요한 정보를 과도하게 저장하지 않는가.
- 이전 결정과 선호를 올바르게 반영하는가.
- 충돌하는 정보를 감지하는가.

### 제안 품질

- 제안이 적절한 시점에 도착하는가.
- 너무 자주 방해하지 않는가.
- 제안 이유가 명확한가.
- 사용자가 실제로 행동할 만한 가치가 있는가.

### 개발 품질

- 요구사항을 올바르게 이해하는가.
- 기존 코드 스타일을 따른다.
- 테스트를 실행하거나 합리적인 검증을 수행한다.
- 회귀를 만들지 않는다.
- 변경 요약이 정확하다.

### 리뷰 품질

- 실제 버그와 리스크를 우선한다.
- 파일/라인 근거를 제시한다.
- 사소한 스타일 지적에 치우치지 않는다.
- 테스트 누락을 잘 찾는다.

### 트레이딩 품질

- 전략 조건을 정확히 계산한다.
- 리스크 제한을 위반하지 않는다.
- 주문 전/후 상태를 검증한다.
- 모든 거래를 기록한다.
- 손실 상황에서 중단 조건을 지킨다.
- 실제 주문 비활성화 상태에서는 주문 후보와 dry-run journal만 생성한다.
- 공식 API가 확인되지 않은 증권사에 실제 주문을 시도하지 않는다.

### 비용 효율

- 같은 정보를 반복해서 대량 프롬프트에 넣지 않는다.
- 요약과 index를 활용한다.
- 작업당 토큰 사용량 추이를 기록한다.
- daily briefing에서 raw source 전체 재투입을 피하고 변경분 중심으로 요약한다.

## 테스트 시나리오 초안

### Memory

- 사용자가 선호를 말하면 wiki에 저장한다.
- 이후 관련 질문에서 해당 선호를 반영한다.
- 사용자가 선호를 바꾸면 이전 선호를 superseded 처리한다.

### Engineering

- 작은 버그를 수정하고 테스트를 통과시킨다.
- PR diff를 리뷰하고 심각도 순으로 지적한다.
- 누락된 테스트를 제안한다.

### Trading

- sandbox 데이터로 전략 신호를 생성한다.
- 주문 제한을 넘는 주문을 차단한다.
- 리스크 정책이 없으면 실제 주문을 차단한다.
- paper trading 주문을 실행하고 로그를 남긴다.
- API 오류 시 실제 주문을 중단한다.
- 삼성증권 connector가 공식 API 없음 상태일 때 자동 주문을 차단한다.
- 토스증권/신한증권 connector는 공식 문서 확인 전 real execution을 비활성화한다.

### Proposal

- 오래 방치된 작업을 감지해 리마인드한다.
- 투자 관심 종목 변화를 감지해 요약한다.
- 진행 중인 개발 프로젝트의 다음 행동을 제안한다.

### Daily Briefing

- 오전 6시 KST briefing job이 실행된다.
- Telegram 요약과 Dashboard 상세가 생성된다.
- 국내 증시 준비와 미국장 마감 요약이 포함된다.
- 개발 우선순위와 pending approval이 포함된다.
- briefing 결과가 memory log에 저장된다.

## 최소 출시 기준

- Markdown memory 구조 동작.
- Agent 작업 로그 기록.
- 개발 작업 수행 및 검증 가능.
- `hermes-agent` 기능 목록 기반 parity checklist 작성.
- 트레이딩은 최소 paper trading 또는 dry-run 모드에서 검증.
- Critical 작업 승인 정책 확정.
- 로컬 daemon, 메신저 봇, 데스크톱 앱의 기본 연결 검증.
- Hermes compatibility MVP checklist 작성.
- daily 6 AM briefing end-to-end 검증.
